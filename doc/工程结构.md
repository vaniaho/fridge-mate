## 二、 文件说明

### 1 `partitions.csv` (分区表)

基于 32MB Nor Flash 优化的自定义分区表，充分利用存储空间：

```
# Name,   Type, SubType, Offset,  Size, Flags
nvs,      data, nvs,     ,        0x6000,
otadata,  data, ota,     ,        0x2000,
phy_init, data, phy,     ,        0x1000,
factory,  app,  factory, ,        4M,
model,    data, fat,     ,        3M,
storage,  data, fat,     ,        12M,
```

| 分区        | 大小   | 用途                           |
| --------- | ---- | ---------------------------- |
| `nvs`     | 24KB | 系统 NVS 存储（含 WiFi/API Key 凭证） |
| `factory` | 4MB  | 应用代码区                        |
| `model`   | 3MB  | 本地 AI 图像分类模型（只读，可独立刷写）       |
| `storage` | 12MB | FAT 文件系统（TTS 音素库、食谱数据、备用存储）  |

### 2. main/ (主程序入口与系统总调度)

#### 2.1 `main/system_events.h`  (通信数据结构)

这是整个团队的**通信契约**，定义全局事件枚举和结构体，每个人都可以 include 这个头文件。

- **`sys_event_type_t`** **枚举**：定义了所有可能的全局事件（如语音指令收到、图像识别完毕、触发休眠等）。
- **Payload 结构体**：当事件发生时，光有类型往往不够，还需要附带数据。因此定义了像 `voice_cmd_payload_t` 和 `vision_infer_payload_t` 这样的结构体。
- **通用外壳** **`sys_event_t`**：使用 `void* payload` 指针，队列中每个元素大小固定且极小。

#### 2.2 `main/task_manager.cpp`

系统的**中央调度室**，包括 FreeRTOS 任务创建、消息队列的总线分发，已集成 `gui_bridge` 桥接调用。

- **事件队列容量**：20 个事件，栈空间 8192 字节
- **`send_system_event`**：全局发送接口，发送方用 `malloc` 分配 payload，总线使用完后自动 `free`
- **`system_bus_task`**：事件大循环 `switch-case`，收到事件后调用对应的业务逻辑和 `gui_bridge` 接口

### 3. `components/system_services/` (系统级公共服务)

将 WiFi 连接、SD 卡驱动、凭证管理等**系统级基础设施**从各业务模块中抽取出来，避免组件间的不必要耦合。

- **`credentials_manager.c`**：基于 NVS 的凭证管理器
  - 首次启动时使用 Kconfig 配置的默认值
  - 运行时可通过 `credentials_set_*()` 接口修改（为 GUI 设置页面预留）
  - 所有修改自动持久化到 NVS
  - 管理的凭证：WiFi SSID/密码、LLM API URL/Key/Model
- **`Kconfig`**：在 `idf.py menuconfig` 中提供 "Smart Fridge Configuration" 菜单
  - WiFi Settings：默认 SSID 和密码
  - LLM API Settings：默认 API URL（豆包）、API Key、模型名称

### 3.1 `components/gui_port/` (显示/触摸/LVGL 任务底座)

GUI 的最底层硬件适配。介于 BSP（`waveshare__esp32_p4_wifi6_touch_lcd_7b`）与业务 UI 之间，封装显示与输入初始化、LVGL 任务调度，是 `gui_app` 与 `gui_bridge` 的运行前提。

- **`gui_port.c`**：调用 `bsp_display_start_with_config()` 完成	LCD 初始化、触摸驱动挂载与 LVGL timer task 启动。关键定制：
  - 将 LVGL 任务栈从默认的 7KB 提升到 32KB，以满足 FreeType 的 `af_autofitter_load_glyph` 在动态渲染中文字形时的栈开销，防止看门狗复位。
  - 使用 BSP 默认的双缓冲与 draw buffer 配置，启动后点亮背光。

### 3.2 `components/gui_port/` 的上层 `components/gui_app/` (LVGL 业务界面层)

基于 LVGL 实现《UI 设计方案》中的 Launcher + 独立应用模块架构，是 `gui_bridge` 抽象接口的具体实现承载层。详见各应用页文件说明：

- **`gui_app.cpp`**：屏幕切换管理器。维护导航栈（`gui_app_id_t` 数组），实现 `gui_app_init()`（Splash → Launcher）、`gui_app_navigate_to(app_id)`（横向滑入动画，按需创建/销毁屏幕）、`gui_app_go_back()`、待机/唤醒。负责 FreeType 字体加载——将 `/internal/NotoSansSC-Regular.ttf` 缓存到 PSRAM 加速渲染，初始化 `font_cn_16/18/24/36` 字号与全局样式。
- **`gui_launcher.cpp`**：桌面启动器。顶部状态栏（设备名 / WiFi 图标 / 实时时钟）、左侧核心信息区（天气卡片 + 留言板）、右侧 4×2 应用网格、底部「拍照存取」横幅入口。
- **`gui_standby.cpp`**：待机/锁屏界面，黑底白字时钟，点击任意位置唤醒。
- **`gui_overlays.cpp`**：全局浮层，挂在 `lv_layer_top()` 上——语音监听指示器、通知弹窗（滑入动画 + 3 秒自动隐藏）、TTS 文本悬浮条（绿色，5 秒自动隐藏）、相机预览/语音助手全屏模态。
- **`app_inventory.cpp`**：库存管理页。侧边栏筛选（全部 / 临期预警 / 水果 / 蔬菜 / 其他）+ Flex 流式食材卡片（名称、分类标签、数量、食用建议），临期卡片加红/黄边框。提供 `app_inventory_refresh()` 供 `gui_bridge_refresh_inventory()` 触发重绘。
- **`app_recipes.cpp`**：智能食谱页。侧边栏（推荐 / 全部）+ 食谱卡片（匹配度、缺失食材数、简要做法），底部 AI 推荐 / 跳转库存按钮。提供 `app_recipes_refresh()`。
- **`app_settings.cpp`**：系统设置页。侧边栏分 5 个 Tab：WiFi 设置（扫描列表 + 密码输入 + 连接状态轮询）、显示设置（亮度滑块）、声音设置（音量滑块）、设备信息、关于。WiFi 扫描与连接通过 `system_manager` 完成。

### 3.3 `components/system_manager/` (运行时系统管理器)

封装运行时可动态控制的系统资源，作为 GUI 设置页与底层 BSP/NVS 之间的桥梁，供 `gui_app` 调用。

- **`system_manager.cpp`**：`SystemManager` 静态类，职责包括：
  - **WiFi**：`scan_wifi_async()` 异步扫描、`connect_wifi(ssid, pass)` 连接（根据扫描结果动态选择认证阈值，支持 OPEN/WPA/WPA2）、状态机管理（含断线自动重连 5 次）、IP/SSID 查询。启动时调用 `wifi_manager_disable_retry()` 接管 `wifi_manager.c` 的重试逻辑。
  - **显示**：`set_brightness(percent)` 通过 BSP PWM 调节背光，NVS 持久化。
  - **声音**：`set_volume(percent)`（硬件实现待补，当前仅持久化）。
  - **设备信息**：`get_firmware_version()` / `get_free_heap()` / `get_total_heap()`。
  - 事件回调更新 WiFi 图标状态（通过 `gui_app_set_wifi_status()`）。

### 4. `components/gui_bridge/` (GUI 事件桥接层)

系统事件与 GUI 框架之间的**抽象桥接层**。已通过 LVGL 接入真实实现：所有接口内部使用 `lv_async_call()` 将后台线程产生的事件安全派发到 LVGL 主渲染线程执行，避免跨 Task 死锁。底层调用 `gui_app` 提供的具体 LVGL 控件操作。

接口包括（见 `gui_bridge.h`）：

- `gui_bridge_wake()` / `gui_bridge_show_standby()`：屏幕亮灭控制（亮屏恢复背光 + 退出待机页；待机关闭背光并切换待机界面）
- `gui_bridge_navigate_to(app_id)` / `gui_bridge_go_back()`：应用页导航
- `gui_bridge_refresh_inventory()`：刷新当前可见的业务页内容（库存页 / 食谱页）
- `gui_bridge_show_notification(title, message)`：顶部通知弹窗（带滑入动画，3 秒自动消失）
- `gui_bridge_show_listening_indicator(bool)`：语音监听状态指示
- `gui_bridge_show_tts_text(text)`：TTS 文本显示（底部绿色悬浮条，5 秒后自动隐藏）
- `gui_bridge_show_camera_preview()` / `gui_bridge_show_voice_assist()`：拍照/语音全屏模态

> 注：桥接层为纯抽象层，不直接持有 LVGL 对象。具体 UI 行为由 `components/gui_app/` 实现，硬件背光/显示控制由 BSP 完成。

### 5. `components/vision_hal/` (图像与视觉模块) 【A】

> **当前状态：待 A 模块接入，工程目录中尚未创建。**

负责摄像头硬件驱动和本地图像推断，是食材识别功能的核心。系统总线已预留 `EVT_VISION_INFER_DONE` 事件及其 `vision_infer_payload_t`（含 `top1_id` 与 `confidence`）挂载点，A 接入后只需投递事件即可被 `task_manager` 路由处理。

### 6. `components/audio_hal/` (语音与音频模块) 【B】

> **当前状态：待 B 模块接入，工程目录中尚未创建。**

负责麦克风采集、本地唤醒词检测和语音播报，是语音交互功能的核心。系统总线已预留 `EVT_WAKE_WORD_DETECTED`、`EVT_VOICE_CMD_RCVD` 事件及 `voice_cmd_payload_t` 挂载点；`EVT_LLM_RESPONSE_READY` 中的 `tts_text` 字段已为 B 的 TTS 播报预留出口（`task_manager` 中标注了 TODO，等 B 的 `audio_api_play_tts()` 就绪后接入）。

### 7. `components/inventory_system/` (本地食材数据库) 【C1】

本地食材库的核心，负责食材数据的增删改查与智能分析。数据持久化依赖 `system_services` 提供的 SD 卡服务；SD 卡不可用时自动降级为纯内存模式。

- **`inventory_manager.cpp`**：
  - 以内存缓存 + SD 卡 JSON 文件作为双层存储（当前方案，后续可迁移至 SQLite）。
  - **数据模型**：采用 `IngredientType`（食材种类）+ `IngredientBatch`（单次存入批次）分离的批次管理架构，支持 FIFO 批次存取。
  - **CRUD 接口**：`add_ingredient`（创建新批次）/ `remove_ingredient`（按 FIFO 优先扣减最早批次）/ `get_all_ingredients` / `clear_ingredient` / `update_ingredient`，每次写操作同步持久化至 `/sdcard/inventory.json`，全程由互斥锁保护并发访问。
  - **过期检测**：`check_expiring_ingredients(threshold_days)` 基于各批次的 `expire_time` 时间戳判断临期/已过期食材。
- **`inventory_history.cpp`**：食材存取历史记录与消耗速率统计。保留 90 天，`compute_all_consumption_rate(days)` 计算日均消耗量，供 `ai_agent` 注入 LLM Prompt。
- **`recipe_matcher.cpp`**：本地食谱库 + 匹配算法。首次运行写入内置默认食谱集到 `/sdcard/recipes.json`，支持 `recipe_match_available()`（完全可做）/ `recipe_match_near(max_missing)`（差 N 样以内）匹配，按覆盖率排序。
- **`category_table.c`**：11 类食材分类表（水果/蔬菜/肉禽/海鲜/蛋奶/豆制品/主食/调味品/饮品/零食/冻品/其他），提供 `category_lookup()` 与默认保质期查询。
- **`include/`**：`inventory.hpp`（数据模型与 CRUD 接口）、`inventory_history.hpp`、`recipe_matcher.hpp`。

### 8. `components/ai_agent/` (云端大模型中枢) 【C2】

负责与云端大模型通信，实现复杂语音指令理解和菜谱推荐等需要 AI 能力的功能。API 凭证由 `system_services` 的 `credentials_manager` 统一管理。

- **`llm_client.cpp`**：基于 `esp_http_client` 封装的 HTTPS 客户端，阻塞式请求云端 LLM API，将响应 JSON 传给 `prompt_builder` 处理。API URL、Key 和模型名称均在运行时从 NVS 动态读取，不硬编码；超时 60 秒兜底。
- **`prompt_builder.cpp`**：
  - `build_llm_request()`：将当前冰箱库存、近 7 天消耗速率、可做菜品推荐注入 System Prompt，与用户文本组装为 OpenAI 兼容格式的完整请求体。模型名称通过 `credentials_get_llm_model()` 动态获取。
  - `parse_and_execute_llm_response()`：解析模型返回的结构化 JSON，提取 `action`（ADD/REMOVE/RECIPE/CHAT）并调用 `inventory_manager` 执行库存操作，同时提取 `tts_reply` 文本封装为 `EVT_LLM_RESPONSE_READY` 事件投递至总线，触发 GUI 刷新与 B 模块播报。

### 9. `components/web_panel/` (Web 控制面板) 【C3】

提供轻量级手机 Web 管理界面，作为原生 App 的替代方案。手机与开发板连接同一 WiFi 后，通过浏览器访问即可使用。

- **`web_server.cpp`**：基于 `esp_http_server` 初始化 HTTP Server，注册路由处理函数；前端静态资源（`index.html` / `style.css` / `app.js`）通过 `_binary_*_start` 符号内嵌编译进固件。WebSocket 推送通过 `web_panel_broadcast_ws()` 对所有连接客户端广播。
- **`api_routes.cpp`**：实现 RESTful API 路由，包括 `GET/POST /api/inventory`（库存查询/录入取出）、`GET /api/history`（存取历史）、`GET /api/status`（系统状态）、`GET/POST /api/settings`（凭证配置，密码脱敏）、`POST /api/chat`（LLM 对话）、`GET /api/recipes/match`、`POST /api/recipes`、`GET/POST /api/notes`（留言板增删）、`GET/POST /api/weather`（天气缓存 + API 配置）、`POST /api/weather/refresh`（手动刷新天气），以及 `/ws` WebSocket 端点。

### 9.1 `components/dashboard/` (桌面看板：留言板 + 天气) 【C5】

承载 Launcher 桌面左侧"天气卡片"与"留言板"的后端逻辑，让两者从硬编码假数据变为真实可管理的数据。

- **`notes_board.cpp`**：留言板。内存缓存 `std::vector<Note>` + 互斥锁 + SD 卡 JSON 持久化（`/sdcard/notes.json`），最多保留 10 条、超出按 FIFO 丢弃。SD 卡不可用时降级纯内存模式。接口：`notes_init/get_all/add/delete/clear`。
- **`weather.cpp`**：天气。从 `credentials_manager` 读取用户配置的天气 API（URL/Key/位置/字段路径），通过 `esp_http_client`（HTTPS + crt_bundle）拉取，用** cJSON 字段路径遍历器**（按 `.` 分隔，如 `now.temp`）提取温度与天气描述，适配和风/心知等不同服务商的 JSON 结构。接口：`weather_init/get/refresh`。
- **`dashboard.cpp`**：后台任务，每 30 分钟（仅 `rtc_time_is_synced()` 时）调用 `weather_refresh()`，成功后通过 `dashboard_set_refresh_callback()` 注册的回调（`main` 设为 `gui_bridge_refresh_dashboard`）通知 GUI 刷新桌面。采用回调注入而非直接依赖 `gui_bridge`，避免组件循环依赖。
- **配置存储**：天气 API 凭证复用 `credentials_manager` 的 NVS key-value 模式（`wx_url/wx_key/wx_city/wx_location/wx_temp_path/wx_text_path`），支持 Web 设置页运行时修改。

### 10. `components/env_sensors/` (环境监测与低功耗) 【C4】

负责外部传感器驱动和系统功耗管理，提升设备使用体验和续航。

- **`rtc_time.c`**：联网时通过 SNTP 服务同步网络时间（北京时间 CST-8），后台周期刷新。NTP 超时不致命——若 RTC 电池维持了有效时间（时间戳 > 2023）则视为已同步；`rtc_time_is_synced()` 供临期检查等模块判断时间可信度。
- **`power_manage.c`**：> **待实现。** 监听 PIR 红外传感器的 GPIO 中断，检测到人体靠近时投递 `EVT_PIR_TRIGGERED` 事件唤醒屏幕；超过 5 分钟无交互则投递 `EVT_GOTO_SLEEP` 事件，驱动 ESP32-P4 进入 Light Sleep 模式（HP 核降频或休眠，LP 核持续轮询传感器）。事件总线与 `gui_bridge` 侧（`gui_bridge_wake` / `gui_bridge_show_standby`）已就绪。

### 11. `web_frontend/` (手机 Web 前端)&#x20;

手机 Web 面板的前端源码，**直接内嵌编译至固件中**，跟随 app 固件一起烧录更新，不占用 SD 卡存储。

- **`index.html`**：单页应用主页面，包含库存列表、录入/取出操作界面和系统设置入口。
- **`style.css`**：响应式样式，适配手机浏览器。
- **`app.js`**：前端逻辑，通过 Fetch API 调用后端 RESTful 接口，通过 WebSocket 接收实时推送通知。

### 12. `internal_fs_data/` (内部文件系统资源)

通过 `main/CMakeLists.txt` 中的 `fatfs_create_spiflash_image(storage ../internal_fs_data FLASH_IN_PROJECT)` 打包进 `storage` 分区（FAT），编译/烧录时自动生成镜像。运行时挂载于 `/internal`。

- **`NotoSansSC-Regular.ttf`**：思源黑体中文字体（约 10MB），由 `gui_app.cpp` 通过 `lv_freetype` 在运行时动态加载，并缓存到 PSRAM 以加速字形渲染。LVGL 默认简陋字库无法满足中文显示，故弃用静态 `.c` 字库改用 FreeType 动态渲染。
