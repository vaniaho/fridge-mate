## 二、 文件说明

### 1 `partitions.csv` (分区表)

基于 32MB Nor Flash 优化的自定义分区表，充分利用存储空间：

```
# Name,   Type, SubType, Offset,  Size, Flags
nvs,      data, nvs,     ,        0x6000,
otadata,  data, ota,     ,        0x2000,
phy_init, data, phy,     ,        0x1000,
factory,  app,  factory, ,        4M,
model,    data, fat,     ,        3M,
storage,  data, fat,     ,        12M,
```

| 分区        | 大小   | 用途                           |
| --------- | ---- | ---------------------------- |
| `nvs`     | 24KB | 系统 NVS 存储（含 WiFi/API Key 凭证） |
| `factory` | 4MB  | 应用代码区                        |
| `model`   | 3MB  | 本地 AI 图像分类模型（只读，可独立刷写）       |
| `storage` | 12MB | FAT 文件系统（TTS 音素库、食谱数据、备用存储）  |

### 2. main/ (主程序入口与系统总调度)

#### 2.1 `main/system_events.h`  (通信数据结构)

这是整个团队的**通信契约**，定义全局事件枚举和结构体，每个人都可以 include 这个头文件。

- **`sys_event_type_t`** **枚举**：定义了所有可能的全局事件（如语音指令收到、图像识别完毕、触发休眠等）。
- **Payload 结构体**：当事件发生时，光有类型往往不够，还需要附带数据。因此定义了像 `voice_cmd_payload_t` 和 `vision_infer_payload_t` 这样的结构体。
- **通用外壳** **`sys_event_t`**：使用 `void* payload` 指针，队列中每个元素大小固定且极小。

#### 2.2 `main/task_manager.cpp`

系统的**中央调度室**，包括 FreeRTOS 任务创建、消息队列的总线分发，已集成 `gui_bridge` 桥接调用。

- **事件队列容量**：20 个事件，栈空间 8192 字节
- **`send_system_event`**：全局发送接口，发送方用 `malloc` 分配 payload，总线使用完后自动 `free`
- **`system_bus_task`**：事件大循环 `switch-case`，收到事件后调用对应的业务逻辑和 `gui_bridge` 接口

### 3. `components/system_services/` (系统级公共服务)

将 WiFi 连接、SD 卡驱动、凭证管理等**系统级基础设施**从各业务模块中抽取出来，避免组件间的不必要耦合。

- **`credentials_manager.c`**：基于 NVS 的凭证管理器
  - 首次启动时使用 Kconfig 配置的默认值
  - 运行时可通过 `credentials_set_*()` 接口修改（为 GUI 设置页面预留）
  - 所有修改自动持久化到 NVS
  - 管理的凭证：WiFi SSID/密码、LLM API URL/Key/Model
- **`Kconfig`**：在 `idf.py menuconfig` 中提供 "Smart Fridge Configuration" 菜单
  - WiFi Settings：默认 SSID 和密码
  - LLM API Settings：默认 API URL（豆包）、API Key、模型名称

### 4. `components/gui_bridge/` (GUI 事件桥接层)

系统事件与 GUI 框架之间的**抽象桥接层**。当前为 Stub 实现（仅日志输出），待 GUI 框架确定（LVGL 或 ESP-Brookesia）后接入真实实现。

接口包括：

- `gui_bridge_screen_on/off()`：屏幕亮灭控制
- `gui_bridge_refresh_inventory()`：刷新食材列表界面
- `gui_bridge_show_notification()`：弹窗通知
- `gui_bridge_show_listening_indicator()`：语音监听状态指示
- `gui_bridge_show_tts_text()`：TTS 文本显示
- `gui_bridge_open_settings()`：打开设置页面

### 5. `components/vision_hal/` (图像与视觉模块) 【A】

负责摄像头硬件驱动和本地图像推断，是食材识别功能的核心。

- **`camera_driver.c`**：基于 ESP32-P4 MIPI-CSI 接口的摄像头驱动，负责图像采集与 ISP 处理（自动曝光、白平衡等）。
- **`image_infer.cpp`**：加载存储在 `model` 分区的本地图像分类模型，运行推断并将 Top-1 类别 ID 和置信度封装为 `vision_infer_payload_t`，通过 `send_system_event(EVT_VISION_INFER_DONE, ...)` 投递至系统总线。
- **`include/vision_api.h`**：向 C 模块暴露的接口，包含触发拍照、启动/停止推断等函数声明。

### 6. `components/audio_hal/` (语音与音频模块) 【B】

负责麦克风采集、本地唤醒词检测和语音播报，是语音交互功能的核心。

- **`wake_word.c`**：离线唤醒词检测逻辑（如"小鲜小鲜"），检测到后投递 `EVT_WAKE_WORD_DETECTED` 事件。
- **`mic_i2s.c`**：通过 I2S 接口驱动 ES7210 回声消除芯片，完成麦克风录音与预处理，为唤醒检测和云端语音识别提供音频数据。
- **`tts_player.c`**：三层 TTS 播报逻辑：Layer 1（预录 PCM 片段）、Layer 2（拼音音素合成）、Layer 3（云端 TTS 流式返回），通过 ES8311 音频编解码芯片驱动扬声器。
- **`include/audio_api.h`**：向 C 模块暴露的接口，包含启动/停止录音、触发 TTS 播报等函数声明。

### 7. `components/inventory_system/` (本地食材数据库) 【C1】

本地食材库的核心，负责食材数据的增删改查与智能分析。数据持久化依赖 `system_services` 提供的 SD 卡服务。

- **`inventory_manager.cpp`**：
  - 以内存缓存 + SD 卡 JSON 文件作为双层存储（当前方案，后续可迁移至 SQLite）。
  - **CRUD 接口**：`add_ingredient` / `remove_ingredient` / `get_all_ingredients`，每次写操作同步持久化至 `/sdcard/inventory.json`。
  - **过期检测**：`check_expiring_ingredients(threshold_days)` 基于 `expire_time` 时间戳判断临期/已过期食材。
  - **待实现**：食材历史记录、消耗速率统计、本地食谱匹配算法。
- **`include/inventory.hpp`**：定义 `IngredientItem` 结构体及全部公共接口，供 `ai_agent` 和 `main` 引用。

### 8. `components/ai_agent/` (云端大模型中枢) 【C2】

负责与云端大模型（豆包）通信，实现复杂语音指令理解和菜谱推荐等需要 AI 能力的功能。API 凭证由 `system_services` 的 `credentials_manager` 统一管理。

- **`llm_client.cpp`**：基于 `esp_http_client` 封装的 HTTPS 客户端，阻塞式请求云端 LLM API，将响应 JSON 传给 `prompt_builder` 处理。API URL、Key 和模型名称均在运行时从 NVS 动态读取，不硬编码。
- **`prompt_builder.cpp`**：
  - `build_llm_request()`：将当前冰箱库存状态序列化注入 System Prompt，与用户语音文本组装为完整请求体。
  - `parse_and_execute_llm_response()`：解析模型返回的结构化 JSON，提取 `action`（ADD/REMOVE/RECIPE/CHAT）并调用 `inventory_manager` 执行，同时提取 `tts_reply` 文本投递给 B 模块播报。

### 9. `components/web_panel/` (Web 控制面板) 【C3】

提供轻量级手机 Web 管理界面，作为原生 App 的替代方案。手机与开发板连接同一 WiFi 后，通过浏览器访问即可使用。

- **`web_server.c`**：基于 `esp_http_server` 初始化 HTTP Server，注册路由处理函数，并通过 WebSocket 向手机端推送实时更新（如临期提醒）。
- **`api_routes.cpp`**：实现 RESTful API 路由，如 `GET /api/inventory`（查询库存）、`POST /api/inventory`（录入/取出食材）等，调用 `inventory_system` 接口处理业务逻辑。

### 10. `components/env_sensors/` (环境监测与低功耗) 【C4】

负责外部传感器驱动和系统功耗管理，提升设备使用体验和续航。

- **`rtc_time.c`**：联网时通过 SNTP 服务同步网络时间，并写入板载 RTC 电池座（CR1220）保持的外置 RTC 芯片，确保断电后食材保质期倒计时不重置。
- **`power_manage.c`**：监听 PIR 红外传感器的 GPIO 中断，检测到人体靠近时投递 `EVT_PIR_TRIGGERED` 事件唤醒屏幕；超过 5 分钟无交互则投递 `EVT_GOTO_SLEEP` 事件，驱动 ESP32-P4 进入 Light Sleep 模式（HP 核降频或休眠，LP 核持续轮询传感器）。

### 11. `web_frontend/` (手机 Web 前端) 【可选】

手机 Web 面板的前端源码，**直接内嵌编译至固件中**，跟随 app 固件一起烧录更新，不占用 SD 卡存储。

- **`index.html`**：单页应用主页面，包含库存列表、录入/取出操作界面和系统设置入口。
- **`style.css`**：响应式样式，适配手机浏览器。
- **`app.js`**：前端逻辑，通过 Fetch API 调用后端 RESTful 接口，通过 WebSocket 接收实时推送通知。
